Gap Docker, Homebrew, Go, CompiledBinaries, ProjectExecutables, NodePackages all lack BDD tests.

The Nix cleaner uses a hardcoded 50MB per generation estimate rather than measuring actual size. Functional but inaccurate — dry-run estimates are misleading.

Gap Should call nix path-info --size for real measurements.

Dry-run prints a warning only instead of estimating. Documented as a Homebrew limitation, but the UX is incomplete — suggests brew cleanup -n instead of running it.

04Totally Fucked Up

Structural problems that actively harm the codebase. These compound with every change.

Split Brain

4 Error Packages With Overlapping Responsibilities

internal/errors/ · internal/pkg/errors/ · internal/adapters/errors.go · pkg/errors/

Four separate error packages compete for the same responsibility. Developers must guess which to import. This is a classic split brain — the type system can't enforce consistency because there's no single source of truth for error handling. Every new file imports a different one, creating an inconsistent patchwork.

Fix Consolidate into ONE error package. Delete the other three. This is the single highest-impact architectural fix.
God Package

internal/domain/ Is a 23-File God Package

internal/domain/ — 23 .go files in one flat package

The domain package contains interfaces, 27 CacheType enums, types, settings, defaults, and macros — all in one package. This violates the single responsibility principle at the package level. Everything depends on it, so it can't evolve independently. It should be split into enums/, operations/, and types/ sub-packages.

Fix Split into domain/enums/, domain/operations/, domain/types/. High effort but pays forever.
Flat Dump

internal/cleaner/ Has 37 Files In One Flat Directory

internal/cleaner/ — 37 source files, no sub-packages

37 cleaner implementations, registry, and factory sit in one flat directory with no domain-based sub-packages. Finding the Docker cleaner means scanning a wall of files. Each cleaner's logic, scanner, executor, and test helpers are interleaved.

Fix Split into per-domain sub-packages: cleaner/nix/, cleaner/docker/, cleaner/golang/, etc. Keep the registry at the top level.

07The #1 Question

The question I cannot answer myself:

"Should the architecture refactors (split domain god package, split cleaner directory, consolidate error packages) happen BEFORE or AFTER adding CI and CLI tests?"

Refactoring without tests is dangerous — we could break behavior we can't detect. But the current structure makes writing tests harder (37-file flat directory, 4 error packages to import correctly). It's a chicken-and-egg problem:

  • Tests first: Safer refactors, but tests are harder to write against the current tangled structure.
  • Refactor first: Cleaner targets for tests, but refactoring untested code risks silent regressions.

My recommendation: set up CI first (task #3), then add CLI integration tests (task #4) against the current structure, THEN refactor with the safety net in place. But I can't decide this for the project — it depends on your risk tolerance and whether you've been running the tool in production.
