06Type Model Issues

27 CacheType string enums without branded types

The CacheType enum in type_safe_enums.go uses string constants. While enum_macros.go provides generic helpers (Parse, Values, IsValid), the CacheType itself is a raw string type — any string can be assigned to it. Branded types would make invalid cache types unrepresentable at compile time.

OperationSettings uses pointer-based optional fields

OperationSettings has 9 pointer fields (*NixSettings, *DockerSettings, etc.) encoding "which operation" via which pointer is non-nil. An interface-based union (type OperationSettings interface { operationSettings() }) would make impossible states unrepresentable — currently you can construct an OperationSettings with all 9 pointers nil or multiple non-nil.

Duration parsing uses string "30d" instead of time.Duration

OlderThan fields across settings use string ("30d") parsed by duration_parser.go at runtime. This is stringly-typed — Go's time.Duration already exists and prevents invalid values at compile time.

09Top Question I Cannot Answer

Should we consolidate to ONE error strategy before or after splitting the god packages?

Migrating fmt.Errorf → cockroachdb/errors touches ~40 call sites across internal/domain/, internal/cleaner/, internal/adapters/. But if we split the domain/ god package first, the import paths change and every file moves. Doing the error migration first means touching files that will move. Doing the split first means the error migration diff is noisier but touches files in their final location.

I lean toward: split first, then migrate errors in the new package locations. The split is structural (moves files, fixes imports), the error migration is semantic (changes error construction within files). Doing structural first gives the semantic changes a stable home. But I'm not certain this ordering minimizes total merge conflicts — it depends on how many files the split touches simultaneously.

Why I can't resolve this: The trade-off depends on how aggressively we split (2 sub-packages vs 5) and whether we use go-modularize patterns. I need direction on target package structure before I can sequence confidently.
