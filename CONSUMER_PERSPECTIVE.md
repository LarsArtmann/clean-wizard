# Clean Wizard — Consumer Perspective Gaps

> What a user (not a developer) would notice is missing, broken, or confusing.

**Date:** 2026-05-27

---

## Critical Gaps (User Can't Do What They Expect)

### 1. No `--version` Flag That Just Prints a Version Number

Users expect `clean-wizard --version` to print something like `v0.1.0`. The version is injected via `fang.WithVersion()` but only shows as part of the help text generated by `fang`. There is no standalone `version` command or a `--version` flag that returns a clean, parseable version string for scripting.

### 2. No Shell Completion Support

Every major CLI tool supports tab completion (bash, zsh, fish, PowerShell). Cobra has built-in support for this, but `clean-wizard` does not register any completion commands. Users who install this tool will immediately notice the lack of tab-completion when typing `clean-wizard clean --mode <tab>`.

### 3. No `man` Pages

There is no generated man page. The `.goreleaser.yaml` does not configure man page generation. Users on Linux and macOS who run `man clean-wizard` will get nothing. Cobra can generate man pages; GoReleaser can package them.

### 4. No `clean-wizard update` or Self-Update Mechanism

Users have no way to check if they're running the latest version or update the tool from within itself. Every modern CLI tool (e.g., `brew`, `nix`, `docker`, `gh`) offers some form of update notification or self-update.

### 5. No Undo / Rollback

There is no mechanism to undo a cleanup operation. If a user accidentally cleans too aggressively, there is no way to recover. Even a simple log of what was deleted (file paths + sizes) would help users understand and potentially manually recover items.

## Significant Gaps (User Experience Pain Points)

### 6. No History / Log of Previous Cleanups

Users cannot see what was cleaned last week, how much space was freed, or which cleaners were run. There is no `clean-wizard history` command and no persistent log file. The config stores `last_clean` as a timestamp but no details about what was done.

### 7. No Scheduling / Automation Built In

The documentation suggests using cron for scheduled cleanups, but the tool itself provides no `clean-wizard schedule` command. Users who want "clean every Sunday at 2am" must manually edit crontab. Compare with tools like `brew autoupdate` or `nix optimise` which can be scheduled declaratively.

### 8. No `clean-wizard doctor` / Health Check

There is no diagnostic command that tells a user:

- Is my config valid?
- Which cleaners are available?
- Which tools are missing?
- Are there permission issues?
- Is my disk dangerously full?

Users must run `scan` and `config validate` separately and piece together the answer.

### 9. No Per-Cleaner Configuration from CLI

Users cannot configure individual cleaners from the command line. Want to change the Nix keep-count from 5 to 3? You must either edit the YAML file manually or run `clean-wizard config edit` and use a text editor. There is no `clean-wizard config set nix.keep_generations 3`.

### 10. No Exclude / Protect Specific Paths on the Fly

The `tempfiles` cleaner supports exclusion patterns in config, but there is no `--exclude` flag on the `clean` command. If a user wants to clean everything except `/tmp/important-data`, they must create or edit a config file first.

### 11. No Size Threshold Filter

Users cannot say "only clean things larger than 100MB" or "skip cleaners with less than 10MB to clean." This is a common use case: "I want to reclaim at least 500MB, tell me which cleaners matter."

### 12. No Confirmation of What Will Be Deleted Before Deletion

The `--dry-run` flag exists, but the actual confirmation prompt just says "Run selected cleaners? Yes/No." It does not show a list of specific files or directories that will be removed. Users must trust the tool or run `--dry-run` separately first.

## Documentation Gaps

### 13. No `CHANGELOG.md` That Users Can Rely On

The CHANGELOG exists but documents internal refactoring (enum consolidation, lint fixes) rather than user-facing changes. A consumer looking for "what changed in the last release" sees developer-oriented entries like "Latent `:=` vs `=` bug" and "49 deprecation warnings across 45+ files."

### 14. No Migration Guide Between Versions

There is no documentation for users upgrading between versions. What changed? Do they need to update their config file? Are any flags renamed or removed?

### 15. README Demo Output Is Fabricated

The README shows demo output (`✓ Homebrew: Freed 245 MB`) that does not match actual tool output format. Users who try the tool and see different output may think something is broken.

### 16. No Troubleshooting Guide for Common Issues

The `HOW_TO_USE.md` has a small troubleshooting section, but it does not cover:

- "Docker daemon is not running" (the most common Docker cleaner failure)
- "Permission denied on /tmp" on Linux
- "No cleanable items found" when the user expects items
- What to do when a cleaner hangs or times out

## Trust & Safety Gaps

### 17. No Verifiable Build Provenance

While `.goreleaser.yaml` configures cosign signing and SBOMs, there is no documentation telling users how to verify the binary they downloaded is authentic. A user who downloads a binary from GitHub Releases has no instructions on running `cosign verify-blob`.

### 18. No Telemetry Opt-In/Opt-Out Notice

The tool does not collect telemetry, but it also does not communicate this to users. Privacy-conscious users have no way to verify that the tool is not phoning home. A simple "This tool does not collect any data" in the README or a `--no-telemetry` flag (even if it's a no-op) builds trust.

### 19. No Dry-Run Report Export

Users cannot export the dry-run results to a file for review or sharing. The `--json` flag works for the `clean` command but there is no `clean-wizard scan --json > report.json` workflow documented, and `scan --json` outputs raw JSON to stdout without saving.

## Distribution & Installation Gaps

### 20. No Pre-Built Binaries on GitHub Releases

The GoReleaser config exists, but the `brews` section has `skip_upload: true`, the Nix section has `skip_upload: auto`, and the Scoop section has `skip_upload: true`. This means:

- `brew install clean-wizard` does not work
- `nix run github:LarsArtmann/nur-packages#clean-wizard` does not work
- `scoop install clean-wizard` does not work

The only installation method is `go install`, which requires the user to have Go installed. Most end-users of a cleanup tool do not have Go.

### 21. No Linux Distribution Packages

There is no `.deb`, `.rpm`, or `.apk` published (GoReleaser is configured to build them but `nfpms` has no repository upload configured). Users on Ubuntu/Debian/Fedora cannot `apt install clean-wizard` or `dnf install clean-wizard`.

### 22. No Windows Support Documentation

The README says macOS. The `.goreleaser.yaml` builds for Windows. There is no documentation for Windows users, no PowerShell examples, and no notes about Windows-specific paths or limitations.

## Polish Gaps (Nice-to-Haves That Competitors Have)

### 23. No Colored/Formatted Output Toggles

There is no `--color=always/never/auto` flag. Users on CI or in log files may get garbled ANSI escape codes. While `NO_COLOR` is likely respected by the Charm libraries, it is not documented.

### 24. No Progress Bar for Long-Running Cleaners

The Docker cleaner or Nix garbage collection can take minutes. There is no progress indication during execution — just a spinner or silence. Users may think the tool has frozen.

### 25. No Quiet Mode

There is no `--quiet` or `-q` flag for CI/automation use cases where only errors should be printed. The `--json` flag helps but still prints non-JSON headers.

### 26. No `clean-wizard status` Command

There is no single command that shows: current disk usage, last cleanup date, available cleaners, and quick recommendations. Users must run `scan`, `config show`, and parse `df` separately.

### 27. No Notification on Completion

For long-running cleanups, there is no option to get a desktop notification or terminal bell when finished. Users who start a cleanup and switch windows have no indication it completed.

### 28. No Config File Generation from Current State

The `init` command creates a default config, but there is no way to generate a config that reflects what the tool discovered about the current system (e.g., "I found Docker, Nix, and Homebrew installed — here's a config optimized for your setup").

### 29. No `--target` or Selective Path Cleaning

Users cannot run `clean-wizard clean --target ~/projects` to only clean items within a specific directory tree. All cleaners operate on hardcoded system paths.

### 30. Profile System Is Not Connected to the Clean Command Properly

The `clean --profile daily` flag exists, but the profile system defines operations by `name` (strings like `"nix-generations"`, `"temp-files"`) that get reverse-mapped to cleaner types. If a profile references a cleaner name that doesn't match exactly, it silently fails with no error. The profile system also doesn't support per-cleaner settings overrides (like `keep_generations: 3` for Nix) from the CLI — they must be in the YAML.

### Top 5 Recommendations (Highest Consumer Impact)

1. **Ship pre-built binaries** — Enable GoReleaser uploads so `brew install`, `nix run`, and direct binary downloads work. Without this, the tool is inaccessible to most users.
2. **Add `clean-wizard doctor`** — A single diagnostic command that shows system state, available tools, config health, and disk usage. This is the first thing a new user should run.
3. **Add cleanup history logging** — Persist what was cleaned, when, and how much was freed. Users need accountability and the ability to review past actions.
4. **Show what will be deleted before confirming** — Replace the generic "Run selected cleaners? Yes/No" with a summary of specific items (file count, total size per cleaner, any high-risk items).
5. **Add `--quiet` and `--color` flags** — Essential for CI/automation users and for users with accessibility needs or non-standard terminals.
